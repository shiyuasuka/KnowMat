2. Strict Physical and Logical Boundaries: Strictly prevent confusion between processing temperatures and testing temperatures. 4. Generalization Guard: Do not overfit to one phrasing pattern or one paper layout. Use body text, tables, captions, and figure descriptions in text. If a quantitative feature is reported but not covered by a whitelist, still extract it with a clear dynamic key and unit hint.

Laboratory-first semantic contract:

- Treat the laboratory schema rooted at `Paper_Metadata` and `items` as the source of truth for meaning, decomposition, and field boundaries.
- Extract the paper title and DOI into root `Paper_Metadata`.
- One lab item corresponds to one unique `Composition_Info + Process_Info` combination with its aligned microstructure and properties.
- When explicit cited benchmark values are reported, extract them as standalone `Reference` items instead of weakening or merging `Target` items.
- The runtime pipeline may serialize the result differently, but semantic correctness is judged against the laboratory item model first.

Item Definition:

- One item = one unique combination of Composition + Process.
- Split into a new item whenever composition, process state, explicit sample/state label, route, build/loading orientation, or parameter level changes.
- Each lab item should carry a stable `Sample_ID`; all tests belonging to the same specimen share that id.
- If the paper defines an abbreviated nomenclature table or sample matrix (for example `3090`, `3067`, `B3090`, `S3090`), enumerate that matrix first and emit one item per explicit row/state. Never keep only one representative row from such a matrix.
- Gradient materials: prefer one item with `Gradient_Material = true` when the paper describes one graded specimen overall.
- Only split a gradient specimen into multiple items when the paper explicitly reports layer-specific or step-specific independent data. In that case, keep `Gradient_Material = true` for each split item and link them with the same `Gradient_Group_ID`.
- In-situ composites: treat matrix and reinforcement as separate items when composition is explicitly defined.
- If the exact same item is tested under multiple test conditions, keep one item and place all those test records inside its `Properties_Info` list.
- If a variant is explicitly named but only sparse evidence is available, keep a sparse item rather than collapsing it into another variant.

Table / provenance safety rules:

- Reconstruct values by header-lock, row-lock, temperature-lock, and unit-lock.
- One table cell maps to one property slot only.
- If ambiguity remains, skip rather than guess.
- Build the full list of sample/state labels and parameter levels before extraction, then verify each has a corresponding item unless evidence is too ambiguous.
- When a process table defines base sample codes and a property table adds direction/state prefixes or suffixes, combine them deterministically (for example process code `3090` plus tensile direction `B/S` means `B3090` and `S3090` are distinct items).
- Never merge properties from different state buckets such as AB vs AA, SLM vs Cast, or different annealing levels.

Composition Extraction (`Composition_Info`):

- `Role` must be `Target` or `Reference`.
- Strictly separate `Nominal_Composition` and `Measured_Composition`.
- `Composition_Type` must be `wt%`, `at%`, or null.
- `Composition_Info.Note` / runtime `composition_note` must capture special annotations such as ranges, upper limits, impurity notes, or balance caveats.
- `nominal_composition` and `measured_composition` must always be flat element->number maps. Never use graded-step keys such as `{"0": {...}, "5": {...}}`; if layer-specific graded data is truly explicit, emit separate runtime compositions instead.
- Faithfully reflect the literature values. Do not rescale or renormalize reported numbers just to force a 100.0 total.
- If sum < 100 and no explicit balance element is given, you may add `other` for the residual balance. If computed `other` is negative, omit it.
- For mature commercial alloys without explicit numeric composition, preserve `Alloy_Name_Raw` and set normalized arrays to null.
- Do not parse trade names, polymers, or commercial aliases into fake element formulas.

Process Extraction (`Process_Info`):

- `Process_Text` must contain both `original` and `simplified`.
- `Process_Text.original` must be as detailed as possible and preserve all explicit process details from the source. Do not simplify or delete process details in `original`.
- `Process_Text.simplified` should keep the key process information, remove redundancy, and remain concise without over-simplifying away important details.
- `Process_Info.Equipment` / runtime `equipment` must capture equipment model, furnace, or manufacturer information when explicitly reported.
- Extract quantified variables into `Key_Params`.
- Prefer canonical keys such as `Laser_Power_W`, `Scanning_Speed_mm_s`, `Layer_Thickness_um`, `Hatch_Spacing_um`, `Build_Plate_Temperature_K`, `Solution_Temperature_K`, `Solution_Time_h`, `Aging_Temperature_K`, `Aging_Time_h`, `Annealing_Temperature_K`, `Annealing_Time_h`, `Cooling_Medium`, `Protective_Atmosphere`, `Powder_Size_avg_um`, `Volumetric_Energy_Density_J_mm3`, `Oxygen_Content_ppm`, `Rolling_Reduction_pct`, `Forging_Temperature_K`, `Extrusion_Ratio`.
- For non-whitelist quantified parameters, still extract dynamic underscore-style keys with unit hints.
- For heat-treatment temperature or time ranges, preserve range strings such as `1273-1373` or `2-4`; never average ranges.
- For multi-stage processing, append stage suffixes such as `_Stage1_K`. Never average distinct stages.

Microstructure Extraction (`Microstructure_Info`):

- `Microstructure_Text` must contain both `original` and `simplified`.
- `Microstructure_Text.original` must be as detailed as possible and preserve all explicit microstructural details from the source. Do not simplify or delete microstructure details in `original`.
- `Microstructure_Text.simplified` should keep the key microstructural information, remove redundancy, and remain concise without over-simplifying away important details.
- `Main_Phase` must describe the matrix phase. Do not label `Laves`, carbides, or gamma-prime as the main phase.
- `Precipitates[]` / runtime `precipitates` must list explicit precipitate types such as `Gamma Prime`, `Laves`, `MC Carbide`, `M2C Carbide`, `M6C Carbide`, or `Sigma Phase`, with volume fraction when reported.
- Extract quantitative fields whenever evidence exists, especially `Main_Phase`, `Porosity_pct`, `Relative_Density_pct`, `Grain_Size_avg_um`, `Precipitate_Size_avg_nm`, `Precipitate_Volume_Fraction_pct`, and `Phase_Fraction_pct`.
- Qualitative statements such as `fully dense`, `full densification`, `pore-free`, or `no porosity detected` do not justify filling `Relative_Density_pct = 100` or `Porosity_pct = 0`. Keep the numeric field null unless the paper gives an explicit number.
- `Relative_Density_pct` must be null when not reported. Never infer 100%.
- Store advanced quantified microstructural features under `Advanced_Quantitative_Features` using explicit dynamic keys.

Properties Extraction (`Properties_Info`):

- Extract all reported mechanical, physical, and chemical properties for the same item.
- Keep strict ordering priority:
  - Tier 1: `Yield_Strength`, `Ultimate_Tensile_Strength`, `Elongation_Total`, `Elongation_Uniform`, `Elongation_At_Fracture`, `Elastic_Modulus`, `Hardness_HV`, `Hardness_HRC`, `Hardness_HB`
  - Tier 2: `Residual_Stress`, `Impact_Toughness`, `Fatigue_Strength`, `Fracture_Toughness`, `Thermal_Conductivity`, `Specific_Heat`, `Coefficient_of_Thermal_Expansion`, `Electrical_Conductivity`, `Electrical_Resistivity`, `Corrosion_Rate`, `Wear_Rate`, `Friction_Coefficient`
  - Tier 3: any additional reported property with a clear dynamic name
- If context is compressive, use `_Compressive` property names and never mix them with tensile names.
- If the paper reports only a generic `Elongation` with no subtype, map it to `Elongation_Total` and record that note explicitly.
- `Test_Temperature_K` is only the actual testing environment temperature. Never place heat-treatment or fabrication temperatures here.
- Also extract `Strain_Rate_s1`, `Tensile_Speed_mm_min`, `Hardness_Load`, `Hardness_Dwell_Time_s`, `Data_Source`, `Test_Specimen`, and `Note` when supported.
